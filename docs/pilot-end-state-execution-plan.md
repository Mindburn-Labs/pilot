# Pilot End-State Execution Plan

Date: 2026-05-12

This is the control artifact for moving Pilot from a governed agent/task prototype to a HELM-governed autonomous startup operating system. It is intentionally conservative: capability claims follow `packages/shared/src/capabilities/index.ts`, `GET /api/capabilities`, and `docs/capabilities.md`.

## Current Observed Repo State

- Shared capability truth exists in `packages/shared/src/capabilities/index.ts` with states, required keys, validation, summaries, and markdown rendering.
- Gateway exposes the read-only registry through `services/gateway/src/routes/capabilities.ts`, mounted at `/api/capabilities` behind the normal `/api/*` authentication path.
- Web exposes `/capabilities` in `apps/web/src/app/capabilities/page.tsx` and reads the API instead of route-local mock state.
- README and roadmap already warn that Pilot is not production-ready as a fully autonomous startup OS.
- Decision Court now splits `heuristic_preview`, `governed_llm_court`, and `unavailable`; governed mode requires HELM-governed model-call receipts and no longer silently falls back to fake adversarial output.
- `operator.computer_use` now supports narrow HELM-governed local safe actions for allowlisted terminal commands, project-scoped file reads/writes, local dev-server status checks, and provider-backed sandbox terminal/file actions with durable `computer_actions` evidence rows. It remains `prototype`, not `production_ready`, because sandbox dev-server status, unrestricted desktop automation, and a passing Safe Computer/Sandbox Action Eval are not complete.

The web `/command-center` surface renders that state without route-local demo data.
- Startup lifecycle templates now compile founder goals into governed lifecycle DAG drafts with required agents, skills, tools, evidence, HELM policy classes, escalation conditions, and acceptance criteria. `/api/startup-lifecycle/persist` stores those DAGs as durable venture, goal, mission, node, edge, and task rows. `/api/startup-lifecycle/missions/:missionId/schedule` identifies dependency-ready nodes and queued task rows. `/api/startup-lifecycle/missions/:missionId/checkpoint` persists a redacted mission/node/edge/task-link snapshot as replayable evidence. `/api/startup-lifecycle/missions/:missionId/recovery-plan` compares the latest checkpoint with current mission state and persists non-executing recovery plan evidence. `/api/startup-lifecycle/missions/:missionId/recover` persists a pre-recovery runtime checkpoint and can reset explicitly failed internal mission nodes to `ready` with recovery-apply evidence without starting execution, rolling back completed work, or touching external systems. `/api/startup-lifecycle/missions/:missionId/rollback` persists a pre-rollback runtime checkpoint and can reopen failed, blocked, or awaiting-approval lifecycle nodes without deleting history or reversing external effects. `/api/startup-lifecycle/missions/:missionId/nodes/:nodeId/execute` can dispatch one scheduled `ready` node through the governed task runtime with mission context and advance newly unblocked pending nodes to `ready`; `/api/startup-lifecycle/missions/:missionId/execute-ready` can execute a bounded batch of currently ready nodes. ## PR Sequence

1. Program truth baseline: keep this execution plan, `docs/capabilities.md`, `docs/pilot-capability-matrix.md`, and `docs/production-readiness-checklist.md` aligned with the shared registry and current blockers.
2. Evidence/audit completion slices: close remaining legacy or non-workspace writer gaps so every meaningful action has durable evidence/audit linkage or explicit non-production blocked state.
3. HELM/RBAC/operator hardening slices: audit remaining non-broker paths, centralize sensitive-route role checks, and pin HELM policy/document versions on meaningful action records.
4. Mission runtime and skills slices: finish long-running checkpoint/retry/replay/resume semantics, bounded parallel agent execution, Tool Broker-governed skills, and proof-DAG handoff continuity.
5. Decision Court production slice: run governed model calls with persisted participants, prompts, costs, receipts, evidence, and refusal semantics when provider/HELM/referee paths fail.
6. Browser production slice: productize scoped read-only active-tab grants and run the YC logged-in extraction eval without exporting credentials.
7. Computer/sandbox production slice: add provider-backed safe command/file/dev-server execution and run the Safe Computer/Sandbox Action Eval with durable proof rows.
8. Tooling/opportunity/PMF slice: finish manifest-backed startup tools, keep stubs unavailable outside demo/test mode, and run PMF/opportunity evals with citations.
9. Command center slice: finish real-state mission graph, agent lanes, evidence drawers, receipt chips, replay viewers, permission graph, and eval status views against durable APIs only.
10. Startup lifecycle and founder-off-grid slices: complete lifecycle templates, controlled recovery, and founder-off-grid boundaries, then run full startup launch, recovery, and off-grid evals.
11. Production promotion slice: promote registry rows only from persisted passing `real_external_eval` metadata; leave failing capabilities non-production with durable blockers.

## Known Blockers

- Decision Court has a durable-proof real-eval verifier for completed governed court runs with bull/bear/referee HELM model-call receipts, costs, evidence, and audit linkage; it still needs an actual passing Decision Court Governed Model Eval run and first-class artifact promotion before it can become `production_ready`.
- Browser execution has scoped read-only sessions, active-tab grants, observation/evidence/replay rows, and a YC browser real-eval verifier for durable proof rows, but still needs a productized browser extension/bridge and an actual passing YC logged-in extraction eval run before it can become `production_ready`.
- Computer use has a narrow safe local execution substrate, provider-backed sandbox command/file execution, Tool Broker routing, command/file evidence, deny policy, and a Safe Computer/Sandbox real-eval verifier; it still needs sandbox dev-server coverage and an actual passing eval run before production-ready claims.
- Subagent lineage has durable root/parent/spawn anchoring, evidence-pack attachment, handoff rows, a proof-DAG API route, and founder-facing proof-DAG inspection, but still needs eval promotion before production-ready claims.
- Approval resume has deterministic parent-history loading and child-row exclusion, but still needs eval promotion before production-ready claims.
- Skills must pass the Skill Invocation Governance Eval with durable brokered invocation, evidence, policy pins, and audit receipts before production-ready claims.
- HELM receipts have mandatory elevated-action sink enforcement, audit-linked receipt evidence, a HELM Governance real-eval verifier for durable proof rows, and Tool Broker blocks elevated tools without HELM policy metadata or evidence persistence, but legacy non-broker execution paths still need equivalent receipt/evidence guards and an actual passing HELM Governance Eval run before production-ready claims.
- Recovery has durable recovery-plan, pre-recovery checkpoint, and recovery-apply evidence with linked audit rows plus a Recovery real-eval verifier, but it still needs an actual passing Recovery Eval run and broader incident/retry coverage before `evidence_ledger` can be production-ready.
- Opportunity scoring needs PMF Discovery Eval promotion and first-class Evidence Center artifact packs before production-ready claims.
- A2A gateway task state is durable in Postgres, and a Multi-Agent Parallel Build real-eval verifier now checks restart-marked A2A threads, ordered messages, multiple handoffs, child runs, and brokered tool evidence, but the actual eval run and broader mission handoff recovery are still required before production-ready claims.
- Command center has a real-eval verifier for durable API fixture evidence, UI screenshot proof, accessibility proof, non-production capability labels, replay surfaces, and audit linkage, but it still needs an actual passing Command Center Real-State UX Eval run and broader mission runtime promotion before production-ready claims.
- Mission runtime has durable venture, goal, mission, node, edge, task, and mission-task rows plus explicit scheduling, checkpoint snapshots backed by manual runtime checkpoint records, pre-recovery/pre-rollback checkpoint records, recovery plan previews, safe failed-node retry reset, constrained internal rollback, bounded ready-node execution, and Full Startup Launch/Domain-to-Deployment real-eval verifiers for completed lifecycle DAG, build/test, DNS/hosting, artifact, deployment, browser, rollback-plan, health, evidence, and audit proof, but still needs actual passing eval runs plus long-running retry/replay/resume semantics before production-ready claims.
- Workspace RBAC remains non-production until HELM Governance Eval promotion. OperatorId scoping now has shared gateway checks, orchestrator runtime rejection, durable denial evidence/audit rows, and a Cross-Workspace Operator Rejection real-eval verifier, but it still needs an actual passing eval run before production-ready claims.
- CI governance check configuration is incomplete: Gitleaks needs the organization `GITLEAKS_LICENSE` secret and Claude async review needs OIDC `id-token: write` workflow permissions. These are merge-process blockers, not eval evidence.
- `control_plane_proof_check` evals are useful regression evidence but are not production promotion evidence. `metadata.executionMode = "real_external_eval"` remains mandatory for production-ready claims and can only be attached by the server-owned trusted eval runner path, not by manual eval-run metadata.
