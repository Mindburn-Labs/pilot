# Pilot v1 — Canonical Spec

**Status:** active — supersedes `docs/audit/v0-beta/*`
**Last updated:** 2026-04-15

## 1. One-Liner

> Pilot is the open-source autonomous founder operating system built on HELM's deterministic execution core, giving a single founder the leverage of a small startup without surrendering control, memory, or execution integrity.

## 2. What Changed from v0-beta

The beta audit in `docs/audit/v0-beta/` targeted a single-process, single-tenant shipping scope with HELM integration deferred. v1 raises the bar on three dimensions:

| Dimension  | v0-beta target                                         | v1 target                                                                                                                |
| ---------- | ------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------ |
| Governance | Local Node.js TrustBoundary                            | **Full HELM sidecar.** Every tool call through Guardian → EffectPermit → signed EvidencePack → ProofGraph.               |
| Surfaces   | Telegram-first, Mini App + web dashboard opportunistic | **All three surfaces at parity.** Telegram bot + Mini App + Next.js web all cover the 5 product modes.                   |
| Tenancy    | Schema-scoped workspace by convention                  | **Multi-tenant SaaS-ready.** Enforced isolation, per-tenant secrets, rate-limit partitioning, per-tenant backup/restore. |

The full implementation plan with phase gates lives at `/Users/ivan/.claude/plans/shimmering-jingling-finch.md` (reflected in this repo as phase markers in code and docs).

## 3. System Boundaries

### 3.1 In Scope (v1.0)

- **5 product modes.** Discover, Decide, Build, Launch, Apply — each end-to-end usable from all three surfaces.
- **HELM integration.** `helm-ai-kernel` v0.4.0 runs as a docker sidecar; Pilot speaks to it over HTTP. Every tool call produces a verifiable receipt.
- **Three surfaces.**
  - Telegram bot (grammY, polling + webhook modes).
  - Telegram Mini App (React Web App inside Telegram).
  - Next.js 15 web dashboard (magic-link auth; Telegram login as alternate).
- **Multi-tenancy.** Workspace boundary enforced at middleware + DB-helper layer; property-based cross-tenant tests; per-tenant secret envelope.
- **Connectors.** GitHub, Gmail, Google Drive, Vercel, DigitalOcean, Cohere (rerank, optional), OpenRouter/Anthropic/OpenAI (LLM providers).
- **Data pipelines.** YC scraper, general startup source scraper (HN/PH/IH/GitHub trending/Reddit startup subs), opportunity clustering, memory ingestion.
- **Observability.** `/metrics` endpoint, Prometheus scrape, Grafana dashboards, OTel traces (optional), structured pino logs.
- **Self-host.** `docker compose up` single-command start; DigitalOcean deployment pathway; per-tenant backup/restore.
- **OSS release.** MIT-licensed; full governance files (LICENSE, CONTRIBUTING, CODE_OF_CONDUCT, SECURITY, CHANGELOG); release pipeline with SBOM.

### 3.2 Out of Scope (deferred past v1.0)

- Multi-region active/active deployment (v1 is single-region per instance).
- Enterprise SSO (SAML/OIDC) — Telegram + magic-link + API key sufficient.
- Policy-pack marketplace — v1 ships with one pack (`founder_ops.v1.json`).
- Policy authoring UI in Studio — v1 authors bundles as JSON files.
- Community connector SDK — v1 ships with bundled connectors only.
- Cross-tenant marketplaces (cofounder candidate discovery, shared research).
- Advanced compliance packs (GDPR/HIPAA/SOC2 overlays) — available from `helm-ai-kernel` but not bundled by default.

## 4. Product Modes (Founder's View)

Each mode is a state the founder's workspace occupies — they can switch between modes, but each mode has its own inputs, actions, artifacts, and gates.

| Mode         | Founder prompt                        | Pilot delivers                                                                                                                                                                                                                         | Receipts                                                                                                     |
| ------------ | ------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------ |
| **Discover** | "I'm curious about AI infra tooling." | Top-10 ranked opportunities with provenance (YC / HN / ProductHunt / GitHub trending / Crunchbase RSS), clustered by market theme, each scored on founder-fit × market-signal × timing × feasibility.                                  | Per-opportunity embed + source link + HELM receipt for each LLM call.                                        |
| **Decide**   | "Run a decision on these three."      | 5-stage court: DocketBuilder → Bull + Bear (parallel) → Referee → ScenarioBuilder → PortfolioSynthesizer. Each stage output is HELM-signed. Cofounder matching and operator recommendation happen here.                                | Signed evidence at each stage; ranked recommendation with confidence.                                        |
| **Build**    | "Start building."                     | Product spec draft → scaffold repo → MVP skeleton → commits via HELM-governed GitHub connector. Iteration loop with spec versioning.                                                                                                   | Every commit produces an EffectPermit + signed receipt; first commit per repo escalates to founder approval. |
| **Launch**   | "Ship it."                            | Deploy to DigitalOcean / Vercel; launch checklist (domain/privacy/terms/analytics); distribution drafting for ProductHunt, HN, Twitter/X, LinkedIn. External posting requires explicit approval.                                       | Per-deploy receipts with health check; per-post receipts on explicit approval.                               |
| **Apply**    | "Apply to YC W2027."                  | Template-based application drafter; per-question generation from concrete evidence (profile section, opportunity, spec version, commit hash, deploy receipt, traction metric); diff-based revision; Playwright submission on approval. | Full submission frozen into a signed evidence pack; sealed PDF stored as artifact.                           |

## 6. HELM Integration (the Vision-Critical Bit)

See `docs/helm-integration.md` (to be written in Phase 10). Summary:

- HELM runs as a sidecar in `infra/docker/docker-compose.yml` (service `helm`).
- Pilot consumes HELM over HTTP on port 8080 (never embedded, never bypassed).
- `packages/helm-client/` is the only place that speaks to HELM.
- On every tool invocation the AgentLoop (`services/orchestrator/src/agent-loop.ts`) calls `helm.evaluate()` before, and `helm.anchor()` after.
- The local `TrustBoundary` in `services/orchestrator/src/trust.ts` becomes a fast-path pre-filter — it can deny, but not allow alone. HELM is authoritative.
- Policy bundle `packs/founder_ops.v1.json` is registered with HELM on startup. It defines operator roles, capability grants, budget envelopes, and policy overlays (external-comms, deploy approval, financial action).
- **Fail-closed semantics.** If HELM is unreachable, every tool call fails. No pass-through. Workspace quarantined after N consecutive failures with owner alert.
- **Evidence fetch.** Founders can retrieve every receipt under `GET /api/governance/receipts/:workspaceId`. The proof graph is browsable at `GET /api/governance/proofgraph/:taskId`.

## 8. Policies and Non-Negotiables

1. **Every tool call goes through HELM.** No bypass, no special-cases. Local fast-path can only deny, never allow alone.
2. **Workspace isolation is enforced at middleware + DB-helper layer, not by convention.** CI fails if any raw SQL omits `workspace_id`.
3. **Connector tokens and founder LLM keys are encrypted per-tenant with HKDF-derived DEKs.** Rotated by documented runbook.
4. **No Redis.** Queues and rate limits live in Postgres via pg-boss and a dedicated token-bucket table. Fewer moving parts, simpler self-host.
5. **Fail-closed defaults everywhere.** Missing config → deny. HELM unreachable → deny. Invalid policy → deny.
6. **No cross-tenant reads, even for admins, without an explicit admin-scoped endpoint that logs and receipts the access.**
7. **Every destructive action (deploy, external post, application submit) escalates through HELM approval.**
8. **Receipts are first-class product surface,** not hidden. Founders see a "Governance" tab with every decision.

4. One external contributor has forked, built locally, and opened a substantive PR that was accepted.

## 9.1 Known Tech Debt Tracked for Phase 10

- **ESLint rollout.** Phase 0 disabled the non-functional `next lint` script in `apps/web` and the missing-toolchain `eslint src/` script in `packages/shared`. Phase 10 (OSS launch) reintroduces a single flat `eslint.config.mjs` at repo root covering every TS/TSX workspace, wires `npm run lint` through turbo, and adds a CI job.
- **Formatting.** Prettier runs via `npm run format` today. Phase 10 adds `npm run format:check` to CI as a blocking gate.

## 9.2 Phase 1 Governance Scope

The production scope now requires both LLM and tool-call governance:

1. HELM proxies `POST /v1/chat/completions` for governed LLM inference and attaches receipt headers.
2. HELM exposes `POST /api/v1/evaluate` for generic tool, deploy, scraping, and external-action decisions, with `/api/v1/guardian/evaluate` as a compatibility alias.
3. Pilot's local `TrustBoundary` can deny or request approval early, but it cannot allow a production tool action by itself.
4. The release gate reads: _every LLM call and every non-finish tool call in production produces a verifiable HELM receipt or fails closed_.

## 10. Open Questions / Known Unknowns

These are recorded here rather than hidden in commits. They do not block v1 unless flagged.

- **LLM cost ceiling at HELM scale.** HELM anchor() calls may add 10–40ms per tool use. If founder-fit scoring runs over 500 opportunities daily per workspace, the HELM round-trip load is material. Benchmark in Phase 1 closing.
- **Playwright submission for YC.** YC's form anti-bot posture is unknown. Phase 7 includes a detection/recovery playbook; worst case, v1 outputs a ready-to-paste markdown that the founder submits manually and Pilot records the submission receipt from the confirmation email.
- **Connector token refresh race.** If a founder's OAuth token expires mid-AgentLoop iteration, we need a clean retry pattern — Phase 6 spec calls this out.
- **Mini App offline mode correctness.** Telegram Web App's service-worker support has platform-specific edge cases; Phase 8 includes a mobile-Chrome + iOS-Safari compatibility matrix.
